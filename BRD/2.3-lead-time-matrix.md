# 2.3. Lead Time Matrix

The Lead Time Matrix defines and manages lead times across Production and Warehouse processes. Lead times are structured hierarchically and can be viewed or configured.

## 1. Production Lead Times – View

Lead times must be displayed in a hierarchical structure:

| Description | Detailed Requirements |
|---|---|
| **Level 1: Branding Departments** | • The system must display a list of all branding departments.<br/>• For each branding department, the following must be displayed:<br/>◦ Department name<br/>◦ Adjustment (in days)<br/>◦ Override (in days)<br/>◦ Personalisation<br/>• Each branding department must be expandable to display branding groups. |
| **Level 2: Categories or Groups** | • Each branding department must contain categories/groups.<br/><br/>Groups are used to group items with similar lead time behaviour based on:<br/>• Print codes<br/>• Product classes or categories<br/>• Specific product codes (base codes only)<br/><br/>For each group, the following must be displayed:<br/>• Name<br/>• Description (auto populated based on specified rules)<br/>• Adjustment<br/>• Override<br/>• Each branding group must be expandable to display quantity breaks. |
| **Level 3: Quantity breaks** | • Each branding group must contain quantity break rules.<br/>Each quantity break must include:<br/>• Lower quantity limit<br/>• Upper quantity limit (or indication that it is infinite)<br/>• Lead time (in days)<br/><br/>**Important note:**<br/>• Quantities refer to the number of branding hits, not the number of physical items. |

### Branding Quantity Display & Calculation Logic

Currently, lead time display is calculated based on the number of products/units ordered. However, this may not accurately reflect production requirements in scenarios where lead times should instead be determined based on the number of branding prints/units required.

To ensure accurate lead time calculation and display:

The system must support lead time calculation based on either:
- Number of units, or
- Number of prints

The applicable calculation method will be determined based on the relevant branding department requirements.

## 2. Production Lead Times – Manage

The system must allow users to configure and manage production lead times at multiple levels, including branding departments, branding groups, and quantity breaks.

### Branding Departments

| Description | Detailed Requirements |
|---|---|
| **List** | The list of branding departments must be managed via the Branding Departments Manager. No departments may be created directly in this section. |
| **Adjustments** | • Users must be able to define a global adjustment per branding department.<br/>• The adjustment must:<br/>◦ Be specified as a number of days<br/>◦ Be added to every lead time rule within that branding department<br/><br/>**Validation rules:**<br/>• Only one adjustment may exist per branding department<br/>• The adjustment value must be numeric<br/>• The adjustment must be stored as a number of days<br/>• If a second adjustment is attempted, the system must prevent the action and display an appropriate validation message |
| **Overrides** | • Users must be able to define a global override per branding department.<br/>• The override must:<br/>◦ Be specified as a number of days<br/>◦ Replace every lead time rule within that branding department<br/><br/>**Validation rules:**<br/>• Only one override may exist per branding department<br/>• The override value must be numeric<br/>• When an override is present, it must replace all calculated lead times within the department<br/>• The system must ensure that override logic takes precedence over all other rules<br/>• If a second override is attempted, the system must prevent the action and display an appropriate validation message |
| **Personalisation** | • Users must be able to define personalisation lead time rules per branding department<br/>Each personalisation rule must include:<br/>• Lower quantity limit<br/>• Upper quantity limit<br/>• Lead time (in days)<br/><br/>**Validation rules:**<br/>• Multiple personalisation rules may exist per branding department<br/>• Quantity ranges must not overlap between rules<br/>• Lower quantity limit must be less than or equal to the upper limit<br/>• Lead time must be numeric<br/>• Duplicate or conflicting quantity ranges must be prevented |

### Branding Groups

| Description | Detailed Requirements |
|---|---|
| **Add** | • Users must be able to create a new branding group within a branding department<br/>Each group must include:<br/>• Group name<br/>• Description (auto populated based on specified rules)<br/>• Grouping rules defined using one or more of:<br/>◦ Prefix filter (e.g. products starting with a specific value)<br/>◦ Print codes<br/>◦ Product classes or categories<br/>◦ Specific product codes (base codes only, not variant codes)<br/><br/>**Validation rules:**<br/>• Group name must be unique within the branding department<br/>• Grouping rules must not overlap with any existing group in the same department<br/>• If overlapping rules are detected, the system must prevent creation and display an appropriate validation message |
| **Delete** | • Superusers must be able to delete an existing branding group |
| **Edit** | • Users must be able to update all fields of an existing branding group<br/><br/>**Validation rules:**<br/>• Updated group name must remain unique within the department<br/>• Updated grouping rules must not overlap with other groups<br/>• If overlap or duplication is detected, the system must prevent the update and display an appropriate validation message |
| **Reorder Groups** | Users must be able to change the order in which branding groups are displayed |
| **Validation** | • The system must validate that no two branding groups within the same department have overlapping:<br/>◦ Prefix filters<br/>◦ Print codes<br/>◦ Product classes or categories<br/>◦ Product codes<br/><br/>• If overlap is detected, the system must prevent saving and display an appropriate validation message |
| **Adjustments** | • Users must be able to define a global adjustment per branding group<br/><br/>**Validation rules:**<br/>• Only one adjustment may exist per group<br/>• Adjustment must be numeric<br/>• Adjustment must apply to all quantity break rules within the group |
| **Overrides** | • Users must be able to define a global override per branding group<br/><br/>**Validation rules:**<br/>• Only one override may exist per group<br/>• Override must be numeric<br/>• Override must replace all lead time rules within the group<br/>• Override must take precedence over adjustments |

### Quantity Breaks

| Description | Detailed Requirements |
|---|---|
| **Add** | Users must define:<br/>• Lower quantity limit<br/>• Upper quantity limit (or specify that it is infinite)<br/>• Lead time (in days) or have the option to specify that it is "On Request"<br/><br/>**Validation rules:**<br/>• Quantity ranges must not overlap within the same branding group<br/>• Lower quantity limit must be less than or equal to the upper limit<br/>• Upper limit must either be numeric or explicitly defined as infinite<br/>• Lead time must be numeric<br/>• Duplicate quantity ranges must not be allowed |
| **Delete** | • Users must be able to delete an existing quantity break<br/><br/>**Validation rules:**<br/>• The system must ensure that deletion does not result in invalid configuration<br/>• If deletion leaves no valid quantity breaks where at least one is required, the system must prevent deletion |
| **Edit** | • Users must be able to update all fields<br/><br/>**Validation rules:**<br/>• Updated ranges must not overlap with existing quantity breaks<br/>• Updated values must remain valid<br/>• Duplicate configurations must not be allowed |
| **Sorting** | • Quantity breaks must always be sorted from lowest to highest quantity |

## 3. Warehouse Lead Times – View

Warehouse lead times must be displayed in a structured and expandable format.

| Description | Detailed Requirements |
|---|---|
| **Level 1: Order Types** | • The system must display a list of all order types.<br/>• For each order type, the following must be displayed:<br/>◦ Order Type Name<br/>◦ Adjustment<br/>◦ Override<br/>• Each order type must be expandable to display groups. |
| **Level 2: Groups** | • Each order type must contain categories/groups.<br/>• Groups are used to group items with similar lead time behaviour<br/>• For each group, the following must be displayed:<br/>◦ Name<br/>◦ Description<br/>◦ Adjustment<br/>◦ Override<br/>• Each group must be expandable to display quantity breaks. |
| **Level 3: Quantity breaks** | Each group must contain quantity break rules.<br/>Each quantity break must include:<br/>1. Lower quantity limit<br/>2. Upper quantity limit<br/>3. Lead time (in hours or days) |

## 4. Warehouse Lead Times – Manage

The system must allow users to configure and manage warehouse lead times at multiple levels, including order types, groups, and quantity breaks.

### Order Types

| Description | Detailed Requirements |
|---|---|
| **List** | Order types cannot be managed via a UI. The current list of order types required for lead times is:<br/>1. Samples<br/>2. Red Slips<br/>3. Replacements – Unbranded<br/>4. Console – Branded<br/>5. Console – Unbranded<br/>6. Unbranded Orders<br/>7. Branches |
| **Adjustments** | • Users must be able to define a global adjustment per order type<br/><br/>**Validation rules:**<br/>• Only one adjustment may exist per order type<br/>• Adjustment must be numeric<br/>• Adjustment must be specified in hours or days<br/>• Adjustment must apply to all lead time rules within the order type |
| **Overrides** | • Users must be able to define a global override per order type<br/><br/>**Validation rules:**<br/>• Only one override may exist per order type<br/>• Override must be numeric<br/>• Override must replace all lead time rules within the order type<br/>• Override must take precedence over adjustments |

### Groups

| Description | Detailed Requirements |
|---|---|
| **Add** | • Users must be able to create a new branding group within an order type<br/>Each group must include:<br/>• Group name<br/>• Description (auto populated based on specified rules)<br/>• Grouping rules defined using one or more of:<br/>◦ Product classes or categories<br/>◦ Specific product codes (base codes only, not variant codes)<br/>◦ Branch<br/><br/>**Validation rules:**<br/>• Group name must be unique within the order type<br/>• Grouping rules must not overlap with any existing group in the same order type<br/>• If overlapping rules are detected, the system must prevent creation and display an appropriate validation message |
| **Delete** | • Users must be able to delete an existing group |
| **Edit** | • Users must be able to update all fields of an existing group<br/><br/>**Validation rules:**<br/>• Updated group name must remain unique within the order type<br/>• Updated grouping rules must not overlap with other groups<br/>• If overlap or duplication is detected, the system must prevent the update |
| **Reorder Groups** | Users must be able to change the order in which groups are displayed |
| **Validation** | • The system must validate that no two groups within the same order type have overlapping:<br/>◦ Prefix filters<br/>◦ Print codes<br/>◦ Product classes or categories<br/>◦ Product codes<br/><br/>• If overlap is detected, the system must prevent saving and display an appropriate validation message |
| **Adjustments** | • Users must be able to define a global adjustment per group<br/><br/>**Validation rules:**<br/>• Only one adjustment may exist per group<br/>• Adjustment must be numeric<br/>• Adjustment must apply to all quantity break rules within the group |
| **Overrides** | • Users must be able to define a global override per group<br/><br/>**Validation rules:**<br/>• Only one override may exist per group<br/>• Override must be numeric<br/>• Override must replace all lead time rules within the group<br/>• Override must take precedence over adjustments |
| **Managing Branches** | • Every branch must exist as a group, with the ability to add quantity breaks per branch.<br/>• Lead times for branches are either specified as a number of days or a specific day of the week. |

### Quantity Breaks

| Description | Detailed Requirements |
|---|---|
| **Add** | Users must define:<br/>• Lower quantity limit<br/>• Upper quantity limit (or infinite)<br/>• Unit (eaches, cases, pallets)<br/>• Production level or production department<br/>• Lead time (in hours or days)<br/><br/>**Validation rules:**<br/>• Quantity ranges must not overlap within the same order type<br/>• Lower limit must be less than or equal to upper limit<br/>• Units must be selected from predefined options<br/>• Lead time must be numeric<br/>• Duplicate configurations must not be allowed<br/>• Duplicate production departments within the same range must not be allowed<br/>• Duplicate production levels within the same range must not be allowed |
| **Delete** | • Users must be able to delete a quantity break<br/><br/>**Validation rules:**<br/>• The system must ensure that deletion does not result in invalid configuration<br/>• If no valid rules remain, the system must prevent deletion |
| **Edit** | • Users must be able to update all fields<br/><br/>**Validation rules:**<br/>• Updated ranges must not overlap<br/>• Updated rules must not duplicate existing configurations<br/>• All values must remain valid |
| **Validation** | The system must prevent:<br/>• Overlapping quantity ranges<br/>• Duplicate combinations of:<br/>◦ Quantity range<br/>◦ Unit<br/>◦ Production level<br/>◦ Production department<br/>• Conflicting rules within the same order type |
