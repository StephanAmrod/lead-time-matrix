# 2. Configuration Components

The following section outlines the interfaces and requirements for managing and configuring lead times.

---

# 2.1. Special Dates Manager

To ensure accurate lead time calculations, the system must account for non-working days such as public holidays, weekends and shutdown periods. A Special Dates Manager is required to maintain and manage these dates.

## 1. Search and Filter Dates

Users must be able to retrieve special dates using the following criteria:

| Filter Criteria | Detailed Requirements |
|---|---|
| **By Year** | Users must be able to filter special dates by year. The system must return all special dates configured within the selected year. |
| **By Category** | Users must be able to filter special dates by category (e.g. public holidays, shutdown periods). The system must return all dates linked to the selected category. |
| **By Name** | Users must be able to search for a special date by name. The search must support partial matching. The system must return all matching results. |

---

## 2. View Dates

| Description | Detailed Requirements |
|---|---|
| **View** | The system must display all special dates based on applied filters, including:<br/>• Date<br/>• Name<br/>• Impacts<br/>• Category |

## 3. Manage Dates

### Manage Special Date Categories

| Description | Detailed Requirements |
|---|---|
| **Add Category** | • Users must be able to create a new category by specifying a category name.<br/><br/>**Validation rules:**<br/>• The category name must be unique.<br/>• Duplicate category names must not be allowed. If a duplicate name is entered, the system must prevent the action and display an appropriate validation message. |
| **Delete Category** | • Users must be able to delete a category.<br/><br/>**Validation rules:**<br/>• A category may only be deleted if no special dates are associated with it.<br/>• If special dates exist for the category, the system must prevent deletion and display an appropriate validation message. |
| **Edit Category** | • Users must be able to update the name of an existing category.<br/><br/>**Validation rules:**<br/>• The updated name must not duplicate an existing category name. If a duplicate name is entered, the system must prevent the update and display an appropriate validation message. |

### Manage Special Dates

| Description | Detailed Requirements |
|---|---|
| **Add Date** | • Users must be able to create a new special date by specifying:<br/>◦ Date (via date selector)<br/>◦ Name (free text)<br/>◦ Impacts (selection of departments impacted by this date)<br/>◦ Category (dropdown selection)<br/><br/>**Validation rules:**<br/>• The date must be unique.<br/>• Duplicate dates must not be allowed. If a duplicate date is entered, the system must prevent creation and display an appropriate validation message. |
| **Delete Date** | • Users must be able to delete an existing special date.<br/><br/>**Validation rules:**<br/>• Only future dates may be deleted.<br/>• If a user attempts to delete a past date, the system must prevent the action. |
| **Edit Date** | • Users must be able to update an existing special date.<br/><br/>**Validation rules:**<br/>• Only future dates may be edited.<br/>• The updated date must remain unique. If a duplicate date is entered, the system must prevent the update and display an appropriate validation message. |
| **Automatic syncs** | • The system must automatically create all South African public holidays annually based on published dates.<br/><br/>**Validation rules:**<br/>• Automatically created dates must not duplicate existing dates.<br/>• If a duplicate is detected, the system must update the existing record.<br/>• Automatically created dates must remain editable. |
