# Story: Lead Time Calculation - I want to calculate jobcard lead times and due dates

**BRD 3.1, 3.2, 3.3**

**BRD References:** 3.1, 3.2, 3.3
**Dev Hours:** 24

---

## Governing Principles

The following principles from **BRD 3.1 (Calculation Model Overview)** and **BRD 3.2 (Core Calculation Principles)** must be enforced as non-functional acceptance criteria for this story:

- Calculation logic must be rule-driven, step-based, deterministic, and extensible
- Lead times must always be calculated at jobcard level first
- All calculations must follow a Base + Additive Components model
- Overrides and adjustments must be applied **before** any add-ons
- Add-ons must be independent, additive, and non-overlapping
- Internal and Client lead times must share the same base calculation and differ only where explicitly defined
- Missing values must default to **TBA**
- All calculations must be deterministic and repeatable
- The design must support adding new components without changing the calculation structure
- All calculations must use the working day calendar (BRD 3.5) and time unit handling (BRD 3.6)

---

## Endpoint Outputs

The endpoint must return, per jobcard:

- **Internal Lead Time**
- **Client Lead Time**
- **Internal Due Date**
- **Client Due Date**

---

## Calculation Pipeline

### Step 1 - Determine Matrix Lead Time

- Retrieve the lead time from the **Production Lead Time Matrix** using the lookup keys: **Branding Department**, **Branding Group**, **Quantity Break**.
- Result: **Matrix Lead Time**.
- If a value cannot be resolved, default the affected output to **TBA**.

> Production-related lead time adjustments occur during a separate downstream process (NPR stage/process) and must not be confused with the current Lead Time Matrix functionality.

### Step 2 - Apply Overrides and Adjustments

Check for an override or adjustment at **Branding Department** level and **Branding Group** level. Apply the following precedence:

1. If an **override** exists → `Base Lead Time = Override` (override takes precedence over all other rules).
2. Else if an **adjustment** exists → `Base Lead Time = Matrix Lead Time + Adjustment`.
3. Else → `Base Lead Time = Matrix Lead Time`.

Result: **Base Lead Time**.

### Step 3 - Determine Add-On Lead Times

Identify all applicable add-on components for the jobcard. Multiple add-ons per jobcard are possible.

| Component | Source Matrix |
|-----------|---------------|
| Console (Warehouse) | Warehouse Lead Time Matrix |
| CMT | Production Lead Time Matrix |
| Personalisation | Production Lead Time Matrix |

Each add-on must:

- Be calculated **independently**
- Use the relevant matrix and rules
- Be looked up using **Branding Department**, **Group**, and **Quantity Break**
- Be applied at most **once per jobcard**
- Be **independent**, **additive**, and **non-overlapping**

Result: **Add-On Lead Times** (collection of values).

### Step 4 - Calculate Total Lead Time (Internal)

```
Total Lead Time = Base Lead Time + Sum of all Add-On Lead Times
```

- If no add-ons apply: `Total Lead Time = Base Lead Time`.
- **Internal Lead Time = Total Lead Time**.

### Step 5 - Calculate Client Lead Time

- If **Console** add-on is **NOT** present: `Client Lead Time = Total Lead Time`.
- If **Console** add-on **IS** present: `Client Lead Time = Total Lead Time + 1 day`.

### Step 6 - Calculate Due Dates

- `Internal Due Date = Approval/Payment Date + Total Lead Time`
- `Client Due Date = Approval/Payment Date + Client Lead Time`

Date arithmetic must use the **working day calendar service (BRD 3.5)** and **time unit handling (BRD 3.6)**:

- When adding lead time, weekends, public holidays, and special dates must be skipped during the addition.
- After applying the lead time, the resulting due date must itself be a valid working day. If it falls on a weekend or a date present in the `SpecialDate` table, advance to the next valid working day. (per TechnicalDesign/04)

> **Action Date** (per Data Dictionary, TechnicalDesign/10) is the standardised term for *Approval/Payment Date* — the date an order is approved and paid for.

## Cross-Cutting Acceptance Criteria

- For jobcards in **approved-and-paid** status, the locked values must be returned (per BRD 3.7).
- For Logo24 jobcards, exception rules apply (per BRD 3.8.1).
- For jobcards on branch orders, branch exception rules apply when calculating the due date (per BRD 3.8.2).
- Lead time matrix changes affecting unapproved/unpaid jobcards must be reflected on the next calculation, respecting the sync behaviour defined in BRD 3.7.

## Implementation Notes (from Technical Design)

### Endpoint Exposure

- The calculation is exposed via a **single GraphQL query** (`productionLeadTime`, see TechnicalDesign/07). Jobcard-level results are produced by this same engine and are invoked internally by the order-level calculation (BRD 3.4) for branded orders.
- Two GraphQL instances are provided:
  - **Internal GraphQL** — authenticated, ops/admin use; supports the flags below.
  - **External GraphQL** — public/partner-facing, safe subset only.
- The API is versioned as `/api/v1/`.

### Inputs

- Either an `orderId`, OR a calculation input object (orderType, brandingDepartments, approvalPaidTimestamps, branchDeliveryMode, etc.).
- Without an Action Date, the response can return only the **lead time in days**. With an Action Date, the response can return an actual **`slaDate`** (calendar date) — the system can only resolve a real date when an Action Date is supplied.

### Internal-Only Flags

- `useStaged: Boolean` (default `false`) — if `true`, calculate against the staged ("blue") matrix containing pending updates. **External callers must not have access to staged data.**
- `includeBreakdown: Boolean` (default `false`) — if `true`, return per-step breakdown:
  - `baseDays`, `addOns[{ code, days }]`, `adjustmentDays`, `overrideApplied`, `calendarShifts[{ date, reason }]`, `finalComputationPath`.
- The breakdown maps directly to the pipeline above; the calculation engine must therefore retain each intermediate value through the pipeline so it can be exposed when requested.

### Response Shape

- **External**: `slaDate` (ISO-8601), `leadTime` (days), `mode` (`'live' | 'staged'`).
- **Internal**: same as external + optional `breakdown` (when `includeBreakdown=true`).

### Source Data Tables (Production Matrix)

- `BrandingDepartment` — department-level `AdjustmentDays`, `OverrideDays`
- `BrandingGroup` — group-level `AdjustmentDays`, `OverrideDays`
- `QuantityBreak` — `LowerLimit`, `UpperLimit`, `LeadTimeDays`
- `PersonalisationRule` — `DepartmentId`, `LowerLimit`, `UpperLimit`, `LeadTimeDays`
- `BrandingByItem` — per-item `AdjustmentDays` / `OverrideDays` where applicable
- `SpecialDate`, `SpecialDateImpact`, `SpecialDateCategory` — for working-day exclusions

### Sync Considerations

- Production matrix changes are staged in **"Update" tables** (Blue) and synced to live (Green) by Flowgear at **02:00 nightly** (per TechnicalDesign/06, 08).
- Live calculations (default mode) read from the live tables; staged calculations read from the update tables (internal callers only).
- A locked jobcard's persisted lead time/due date must be retained on the jobcard so subsequent matrix updates do not affect the stored result. The persisted breakdown should be retained where feasible to support audit.

### Non-Functional Requirements (per TechnicalDesign/11)

- **Response time SLA**: < **250ms** at the 99th percentile.
- **Throughput**: must scale to **10 RPS** under peak load.
- **Caching**: cache calculation inputs; cache must be cleared at night when the matrix sync runs.
- **Observability**: every administrative adjustment that affects calculation inputs must produce a structured log entry (`UserId`, `Timestamp`, `EntityId`, `ChangeDelta` before vs after) and be forwarded to **Signoz**.
