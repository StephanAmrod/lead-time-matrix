# Story: Lead Time Calculation - I want to calculate order lead times for branded and unbranded orders

**BRD References:** 3.1, 3.2, 3.4
**Dev Hours:** 20

---

## Governing Principles

The following principles from **BRD 3.1 (Calculation Model Overview)** and **BRD 3.2 (Core Calculation Principles)** must be enforced as non-functional acceptance criteria for this story:

- Calculation logic must be rule-driven, step-based, deterministic, and extensible
- Lead times must always be calculated at jobcard level first
- All calculations must follow a Base + Additive Components model
- Overrides and adjustments must be applied **before** any add-ons
- Add-ons must be independent, additive, and non-overlapping
- Internal and Client lead times must share the same base calculation and differ only where explicitly defined
- Missing values must default to **TBA**
- All calculations must be deterministic and repeatable
- The design must support adding new components without changing the calculation structure
- All calculations must use the working day calendar (BRD 3.5) and time unit handling (BRD 3.6)

---

## Endpoint Outputs

The endpoint must return, per order:

- **Internal Order Lead Time**
- **Client Order Lead Time**

---

## Step 1 - Determine Order Type

- **Branded** — the order has at least one branding jobcard. The branded path invokes the jobcard calculation (per BRD 3.3) for each jobcard, then aggregates per Step 2A.
- **Unbranded** — the order is **stock-only** (zero jobcards). The unbranded path reads directly from the warehouse matrix per Step 2B.

> **Action Date** (per Data Dictionary, TechnicalDesign/10) is the standardised term for *Approval/Payment Date* — the date an order is approved and paid for.

---

## Step 2A - Branded Orders (Jobcard Aggregation)

Use the calculated **jobcard internal and client lead times** (per BRD 3.3 / [3.3-calculate-jobcard-lead-time.md](3.3-calculate-jobcard-lead-time.md)) for aggregation.

| Scenario | Internal Order Lead Time | Client Order Lead Time |
|----------|--------------------------|------------------------|
| Single jobcard | Jobcard Total Lead Time | Jobcard Client Lead Time |
| Merged jobcards - same branding department | Max(Jobcard Total Lead Time) | Max(Jobcard Client Lead Time) |
| Merged jobcards - different branding departments | Max(Jobcard Total Lead Time) + 1 day | Max(Jobcard Client Lead Time) + 1 day |

---

## Step 2B - Unbranded Orders

The pipeline mirrors the jobcard pipeline but uses the **Warehouse Lead Time Matrix**.

1. **Determine Matrix Lead Time** - Retrieve from the Warehouse Lead Time Matrix using **Order Type**, **Group**, and **Quantity Break**.
2. **Apply overrides/adjustments** to derive Base Lead Time using the same precedence as BRD 3.3 Step 2 (override > adjustment > matrix only).
3. **Determine add-on lead times** (e.g. Console from the warehouse matrix). Multiple add-ons per order are possible.
4. `Internal Order Lead Time = Base Lead Time + Sum of Add-On Lead Times`
5. `Client Order Lead Time = Internal Order Lead Time`
6. **Apply Action Date and validate the final due date** — using the working day calendar (BRD 3.5), add the lead time to the Action Date. The resulting due date must be a valid working day; if it falls on a weekend or a date in the `SpecialDate` table, advance to the next valid working day. (per TechnicalDesign/04)

> For unbranded orders (including console orders) **no additional client approval lead time** is added; the `+1 day` Console rule applies exclusively to branded jobcard rules (per BRD 3.3 Step 5).

---

## Cross-Cutting Acceptance Criteria

- Where an order contains one or more **locked jobcards** (approved and paid), aggregation must use the locked jobcard values (per BRD 3.7) — the order calculation must NOT recompute those jobcard values from the current matrix.
- Lead time matrix changes affecting unapproved/unpaid orders must be reflected on the next calculation, respecting sync behaviour (per BRD 3.7).
- Logo24 (BRD 3.8.1) and branch order (BRD 3.8.2) exceptions must be applied where they affect order-level results.

## Implementation Notes (from Technical Design)

### Endpoint Exposure

- Order-level calculation is the **primary surface** of the calculation engine, exposed via a single GraphQL query (`productionLeadTime`, see TechnicalDesign/07). For branded orders this internally invokes the jobcard calculation (BRD 3.3 / [3.3-calculate-jobcard-lead-time.md](3.3-calculate-jobcard-lead-time.md)) per jobcard.
- Two GraphQL instances are provided:
  - **Internal GraphQL** — authenticated, ops/admin use; supports the flags below.
  - **External GraphQL** — public/partner-facing, safe subset only.
- The API is versioned as `/api/v1/`.

### Inputs

- Either an `orderId`, OR a calculation input object (orderType, brandingDepartments, approvalPaidTimestamps, branchDeliveryMode, etc.).
- Without an Action Date, the response can return only the **lead time in days**. With an Action Date, the response can return an actual **`slaDate`** (calendar date).

### Internal-Only Flags

- `useStaged: Boolean` (default `false`) — if `true`, calculate against the staged ("blue") matrix containing pending updates. **External callers must not have access to staged data.**
- `includeBreakdown: Boolean` (default `false`) — if `true`, return per-step breakdown:
  - `baseDays`, `addOns[{ code, days }]`, `adjustmentDays`, `overrideApplied`, `calendarShifts[{ date, reason }]`, `finalComputationPath`.

### Response Shape

- **External**: `slaDate` (ISO-8601), `leadTime` (days), `mode` (`'live' | 'staged'`).
- **Internal**: same as external + optional `breakdown` (when `includeBreakdown=true`).

### Source Data Tables

- **Branded path** delegates to the jobcard calculation, which reads the production matrix tables (see [3.3-calculate-jobcard-lead-time.md](3.3-calculate-jobcard-lead-time.md)).
- **Unbranded path** reads:
  - `OrderType` — order-type-level `AdjustmentDays`, `OverrideDays`
  - `WarehouseGroup` — group-level `AdjustmentDays`, `OverrideDays`
  - `WarehouseQuantityBreak` — `LowerLimit`, `UpperLimit`, `LeadTimeDays`, optional `DepartmentId` for production-department-specific add-ons
- `SpecialDate`, `SpecialDateImpact`, `SpecialDateCategory` — for working-day exclusions and final-date validation.

### Sync Considerations

- **Production matrix** changes are staged in "Update" tables (Blue) and synced to live (Green) by Flowgear at **02:00 nightly** (relevant to branded jobcard aggregation).
- **Warehouse matrix** changes are applied **immediately** in real time, transaction-safe (relevant to unbranded orders and Console add-ons).
- Live calculations (default mode) read from the live tables; staged calculations read from the update tables (internal callers only).

### Non-Functional Requirements (per TechnicalDesign/11)

- **Response time SLA**: < **250ms** at the 99th percentile.
- **Throughput**: must scale to **10 RPS** under peak load.
- **Caching**: cache calculation inputs; cache must be cleared at night when the matrix sync runs.
- **Observability**: every administrative adjustment that affects calculation inputs must produce a structured log entry (`UserId`, `Timestamp`, `EntityId`, `ChangeDelta` before vs after) and be forwarded to **Signoz**.

### Amtrack Integration & Golden Record (per TechnicalDesign/02, 08, 11)

- Moyo is the **Golden Record** / single source of truth for lead time calculations.
- Amtrack must call this endpoint and **must NOT perform internal lead time calculations** any longer.
- Amtrack consumers must implement a **circuit-breaker pattern**: if responses exceed **1500ms** or fail **3 times consecutively**, Amtrack falls back to its local read-only database replica until recovery.
- Sync directionality is **unidirectional Moyo → Amtrack**; data changes in Moyo must not sync back into Amtrack for lead time calculations.
